From Recommendation to Campaign Generation

Standard Rec

"Which items should this user see?"

\(f(\text{user}) \to \text{ranked items}\)

Campaign Targeting

"Which users should see this campaign?"

\(g(\text{campaign}) \to \text{ranked users}\)

GenRec enables a third paradigm:
Generate user interest profiles as structured tokens → match against campaign criteria.
No need to score every (user, campaign) pair — just decode and match.

Campaign scoring:

\[ \text{score}(u, c) = \sum_{l \in \text{levels}} \mathbb{1}\left[\hat{t}_u^{(l)} \in \mathcal{T}_c^{(l)}\right] \cdot w_l \]

where \(\hat{t}_u^{(l)}\) = user's generated token at level \(l\), \(\mathcal{T}_c^{(l)}\) = campaign's target set

Instead of "will this user click this item?", GenRec answers "what does this user want next?" — and campaigns simply filter on the answer.

Interpretable Campaign Matching

Campaign defines targeting criteria via token levels:
• L1 (Category): Electronics
• L4 (Brand): Apple
• L6 (Intent): premium / audiophile

User's generated token sequence reveals preferences:

\[ \hat{\mathbf{t}}_u = [\underbrace{\text{Electronics}}_{L1}, \underbrace{\text{Apple}}_{L3/L4}, \underbrace{\text{high}}_{L5}, \underbrace{\text{premium}}_{L6}, \underbrace{\text{ANC, over-ear}}_{L7}] \]

Matching = token overlap:

\[ \text{match}(u, c) = \frac{|\hat{\mathbf{t}}_u \cap \mathcal{T}_c|}{|\mathcal{T}_c|} \]

Example:

CampaignTarget TokensUser GeneratesMatch?
"Premium Apple Audio"{Electronics, Apple, premium}{Electronics, Apple, premium, ANC}✓ 3/3
"Budget Samsung Mobile"{Electronics, Samsung, budget}{Electronics, Apple, premium, ANC}✗ 1/3
Every matching decision is fully interpretable — you can explain exactly WHY a user was targeted: "they're predicted to want premium Apple audio products next."

Cold-Start Item Recommendation

The Problem

New item arrives with attributes but no interaction history

  • ID-based models: cannot score it at all
  • Collaborative filtering: no co-occurrence data
  • Must wait for warm-up period

GenRec Solution

New item gets tokens from its attributes:

L1=Electronics, L4=Sony, L5=mid, L6=casual, L7=wireless

Instant scoring — no retraining!

Scoring a cold-start item \(i_{\text{new}}\) for user \(u\):

\[ P(i_{\text{new}} | u) = \prod_{l=1}^{L} P\!\left(t_{i_{\text{new}}}^{(l)} \,\big|\, \mathbf{h}_u, \, t_{i_{\text{new}}}^{(1:l-1)}\right) \]

The model estimates how likely it is to generate this item's tokens given the user's history.

Key advantages: (1) Zero warm-up time, (2) No retraining needed, (3) Works from day one of item launch
If a user frequently generates "Apple, premium, headphones" tokens, a new Apple headphone gets scored highly even with zero interactions — because it aligns with the user's generative profile.

Generation-Augmented Retrieval (GAR)

Invert the pipeline: Instead of scoring existing candidates, generate candidate descriptions first, then resolve against catalog.

Standard retrieval: query → ANN search over item embeddings → candidates

GAR: query → LLM generates item descriptions → resolve via ANN → candidates

\[ \text{GAR}(q) = \text{ANN}\!\left(\text{encode}\!\left(\text{LLM}(q)\right), \; \mathcal{I}\right) \]

Connection to GenRec: GenRec tokens are a structured version of GAR — instead of free-text generation, generate in a controlled token vocabulary with guaranteed catalog resolution.
GAR shows that generation-then-retrieval outperforms retrieval-only. GenRec's hierarchical tokens make this structured and controllable — every generated sequence maps to real items.

StaticCSR — YouTube Constrained Sequential Retrieval (2026)

Problem: Production video recommendation needs freshness, diversity, and business rules — standard beam search ignores all of these.

Key idea: Build static constraints INTO the generation process:

\[ P(i | u, \mathcal{C}) = P(i | u) \cdot \prod_{c \in \mathcal{C}} \phi_c(i) \]

where \(\phi_c(i)\) are constraint satisfaction indicators:

• Freshness: \(\phi_{\text{fresh}}(i) = \mathbb{1}[\text{age}(i) < \tau]\)

• Diversity: \(\phi_{\text{div}}(i) = \mathbb{1}[i \notin \text{recent\_categories}]\)

• Policy: \(\phi_{\text{policy}}(i) = \mathbb{1}[i \in \text{allowed\_set}]\)

Architecture:

  • Sequential encoder (Transformer) for user history
  • Constrained beam search: prune beams violating constraints at each step
  • Static constraints compiled once → applied at inference without re-scoring
StaticCSR proves that generative retrieval can serve billions of users at YouTube scale — the trick is making constraints zero-cost by pre-compiling them into the decoding trie.

Emerging Directions (2025–2027)

(a) LLM-Native Recommendation

Claude/GPT generating product recommendations directly from text descriptions. No separate rec model — the LLM IS the recommender.

(b) Multi-Modal Generative Retrieval

Images + text + behavior jointly tokenized. Generate visual tokens alongside semantic ones. Product thumbnails as retrieval signals.

(c) Real-Time Personalized Generation

Generate recommendations conditioned on live context: time, location, current cart, browsing session. Streaming inference with KV-cache.

(d) Diffusion Models for Rec

DiffuRec, DreamRec: model user preference as a continuous distribution, "denoise" toward ideal next item. Handles uncertainty naturally.

Diffusion formulation:

\[ p_\theta(x_0 | u) = \int p(x_T) \prod_{t=1}^{T} p_\theta(x_{t-1} | x_t, u) \, dx_{1:T} \]

Gradually "denoise" from random point in item embedding space toward user's ideal next item.

The field is splitting: (a) LLMs as zero-shot recommenders for simple cases, (b) specialized generative models for complex production systems. Both directions are active.

Open Problems in Generative Recommendation

1. Accuracy Gap vs. Discriminative Models

SASRec/BERT4Rec still win on pure ranking metrics. GenRec trades accuracy for controllability, interpretability, and cold-start ability. Can we close this gap?

2. Online Serving Latency

Beam search decoding: \(O(B \times L \times V)\) where B=beam width, L=levels, V=vocab per level

vs. embedding lookup + ANN: \(O(\log N)\)

Production requirement: <50ms p99 for retrieval stage

3. Evaluation Standards

ProtocolWhat it measuresProblem
uni100Rank 1 positive among 99 random negativesInflated metrics; easy negatives
Full-catalogRank among all itemsExpensive; hurts generative models more
Online A/BReal engagement/revenueExpensive to run; few academic labs can

4. Catastrophic Forgetting

Token vocabularies evolve (new brands, categories). Continual learning without forgetting previously learned user preferences remains unsolved.
The field is productive but honest about limitations. These four problems define the research agenda for 2026-2028.

Your GenRec System — Problem Statement

Domain

  • 45K items — wireless electronics (Amazon.in)
  • 842K sequences — behavioral histories
  • 90K customers — active buyers + explorers
  • Goal: personalized campaign targeting + next-item prediction

Dual-Stream Architecture

Purchase stream: items the user actually bought (high signal, sparse)
Explore stream: items viewed/clicked/added-to-cart (lower signal, dense)

Objective: Given user history \(\mathbf{s}_u = [s_1, s_2, \ldots, s_n]\) (interleaved purchase + explore),

generate the token sequence \(\hat{\mathbf{t}}_{n+1} = [t^{(1)}, t^{(2)}, \ldots, t^{(L)}]\) of the next item.

\[ \mathcal{L} = -\sum_{l=1}^{L} \log P\!\left(t_{n+1}^{(l)} \,\big|\, \mathbf{s}_u, \, t_{n+1}^{(1:l-1)}\right) - \lambda \cdot \mathcal{L}_{\text{ID}} \]
This isn't a toy benchmark — it's a real production system with 2M catalog items, sparse purchase data, and the business requirement of "explain why this user was targeted."

Hierarchical Token Schema

7-level hierarchy (L1 → L7d):

L1Product Family18Electronics
L3Ecosystem~80Apple, Samsung, …
L4Brand~350Apple, Sony, JBL, …
L5Price Tier4budget / mid / high / premium

Design properties:
• 90% uniqueness: token sequences uniquely identify items
• 2M catalog coverage via OPQ codes at L7
• Coarse→fine: L1-L6 are interpretable, L7 handles fine-grained discrimination
• No RQ-VAE: interpretable tokens chosen over learned codes
The token schema IS the secret sauce. Interpretable levels (L1-L6) give you campaign targeting for free. OPQ levels (L7) recover the fine-grained item identity that interpretability sacrifices.

GenRec v17 Architecture

Encoder: Bidirectional Transformer

\[ \mathbf{H} = \text{BiTransformer}\!\left(\mathbf{E}_{\text{item}} + \mathbf{E}_{\text{pos}} + \mathbf{E}_{\text{level}}\right) \]

• Per-level embeddings: separate embedding per token level (L1-L7)

• Dual positional encoding: absolute position + level-within-item position

Decoder: Causal cross-attention

\[ P(t^{(l)} | \mathbf{H}, t^{(1:l-1)}) = \text{softmax}\!\left(W_l \cdot \text{CrossAttn}(t^{(1:l-1)}, \mathbf{H})\right) \]

• Autoregressive: each level conditioned on all previous levels

• Multi-position training: predict at every sequence position (not just last)

Item ID Head: Weight-tied classification + InfoNCE

\[ \mathcal{L}_{\text{ID}} = -\log \frac{\exp(\mathbf{h}_u^T \mathbf{e}_{i^+} / \tau)}{\sum_{j} \exp(\mathbf{h}_u^T \mathbf{e}_j / \tau)} \]

• Resolves token collisions (multiple items sharing same L1-L6 tokens)

• Weight-tied: shares item embeddings with encoder input layer

The architecture is encoder-decoder with a twist: the ID head acts as a "collision resolver" — when two items share the same semantic tokens, it discriminates using learned item embeddings.

Lessons Learned

1. Token quality > architecture complexity
Upgrading from random OPQ to curated L1-L6 + fine-tuned OPQ gave more lift than any architecture change. Garbage tokens in → garbage predictions out.
2. Interpretable tokens sacrifice accuracy for controllability
Learned RQ-VAE codes compress better but are opaque. Interpretable tokens let you constrain, explain, and target — worth the accuracy trade-off for this use case.
3. DPO shifts generation toward purchases
Direct Preference Optimization: treat purchases as "chosen" and explores as "rejected." Result: model generates tokens for items users actually buy, not just browse.
4. Jointly-trained ID head is essential
Without it, token collisions kill accuracy. With it, the model resolves ambiguity (multiple items with same L1-L6) using learned item embeddings.
5. Campaign targeting = killer app
Pure next-item accuracy is where discriminative models excel. But "which users match this campaign?" is something only GenRec can answer interpretably at scale.
The biggest lesson: don't compete with SASRec on its home turf (ranking). Compete where generative models have structural advantages — controllability, interpretability, cold-start, and novel applications like campaign targeting.

What You'd Do Differently

Architecture

Learned edge weights:
Currently tokens within an item are equally weighted. Better: learn attention weights between levels — "brand matters more than price tier for this user."
Attention-based cluster assignment:
Replace hard OPQ quantization at L7 with soft attention over cluster centroids. Differentiable end-to-end.

Process

Online A/B testing earlier:
Offline metrics (HR@10) don't predict online performance well. Should have instrumented A/B infrastructure from v2, not v15.
Temporal decay in sequences:
A purchase 6 months ago shouldn't weigh the same as yesterday's. Exponential decay or learnable positional weights per recency bin.

Soft cluster assignment (proposed):

\[ t_i^{(L7)} = \sum_{k=1}^{K} \alpha_k \cdot c_k, \quad \alpha_k = \frac{\exp(\mathbf{q}_i^T \mathbf{c}_k / \tau)}{\sum_{j} \exp(\mathbf{q}_i^T \mathbf{c}_j / \tau)} \]

where \(\mathbf{c}_k\) are learnable cluster centroids and \(\alpha_k\) are soft assignments.

Hindsight is 20/20 — but the core theme is: make everything differentiable (soft assignments, temporal weights) and validate online earlier.

Quiz — Chapter 7 (Part 1)

Q1: In GenRec campaign targeting, what makes it possible to explain WHY a user was matched to a campaign?
A: The interpretable token schema (L1-L6). Each level has human-readable meaning (category, brand, intent), so the match is explained as token overlap: "User generates [Electronics, Apple, premium] which matches the campaign's target set."
Q2: How does GenRec handle cold-start items that have zero interaction history?
A: The new item is assigned tokens from its attributes (L1-L7). The model scores it by computing the probability of generating those tokens given the user's history: \(P(i_{\text{new}} | u) = \prod_l P(t_i^{(l)} | \mathbf{h}_u, t_i^{(1:l-1)})\). No retraining required — works from day one.
Q3: What is Generation-Augmented Retrieval (GAR), and how did it perform at Amazon?
A: GAR inverts the standard pipeline: instead of searching over item embeddings directly, an LLM first generates candidate descriptions, which are then resolved against the catalog via ANN. At Amazon, GAR produced +$209M incremental OPS and +12% recall improvement. GenRec's structured tokens are a controlled version of the same idea.

Quiz — Chapter 7 (Part 2)

Q4: In your GenRec system, why does the ID head exist alongside the token decoder? What problem does it solve?
A: Token collisions — multiple items can share the same L1-L6 tokens (e.g., several Apple premium headphones). The ID head uses InfoNCE loss with weight-tied item embeddings to discriminate between items that share semantic tokens, resolving the ambiguity that pure token generation cannot.
Q5: Your GenRec v5 achieves HR@10 = 0.503 vs SASRec's 0.706. Given this gap, what is the business justification for using GenRec?
A: GenRec offers capabilities that discriminative models fundamentally cannot: (1) interpretable predictions for campaign targeting, (2) zero-shot cold-start scoring, (3) constrained generation at zero inference cost, and (4) automated campaign composition via token overlap. The 29% accuracy gap is the cost of these structural advantages — and for campaign targeting use cases, there IS no discriminative alternative.
