Chapter 5

LLM Training Techniques Applied to
Generative Recommendation

Teacher forcing, scheduled sampling, lookahead, RLHF, DPO, contrastive learning, distillation, LoRA, speculative decoding, and diffusion

Why LLM Techniques Transfer to GenRec

Shared Foundation:

GenRec = autoregressive token generation

Same decoder architecture, same left-to-right factorization

Shared Challenges:

  • Exposure bias (train vs inference mismatch)
  • Mode collapse (generating only popular items)
  • Reward optimization (aligning with user preferences)

Key Difference:

LLMs: free-form text tokens (vocab ~50K)
GenRec: discrete structured outputs — item tokens from a hierarchical codebook (vocab ~256 per level)

Applicable Techniques:

All of autoregressive training + alignment methods transfer directly

If you can train GPT to generate coherent text, the same tricks help generate coherent item token sequences. The vocabulary is smaller but the structure is stricter.

Teacher Forcing and Exposure Bias

Teacher Forcing Loss:

\[ \mathcal{L} = -\sum_{t=1}^{K} \log P(c_t \mid c_1, \ldots, c_{t-1}, \text{ctx}) \]

At training: condition on ground-truth prefix \(c_1, \ldots, c_{t-1}\)

The Problem at Inference:
Model feeds its own predictions — never saw its own mistakes during training!

Exposure Bias in GenRec:

  • If L1 prediction is wrong, all subsequent levels drift
  • Model never practiced recovering from errors at upper levels
  • Compounding: error at level \(k\) propagates to levels \(k+1, \ldots, K\)
It's like learning to drive only on a rail track, then being thrown onto open roads. Teacher forcing never lets the model experience its own imperfect decisions.

Scheduled Sampling

Idea: gradually replace ground-truth with model's own predictions

At step \(t\), use:

\[ c_t^{\text{input}} = \begin{cases} c_t^{\text{true}} & \text{with prob } (1 - \varepsilon_t) \\ \arg\max P(c_t \mid \ldots) & \text{with prob } \varepsilon_t \end{cases} \]

Schedule: \(\varepsilon_t\) increases from \(0 \to \varepsilon_{\max}\) over training

Curriculum:

Early training: stable (teacher forcing, ε≈0)
Late training: realistic (self-feeding, ε≈0.3-0.5)

Benefits & Limitations:

  • ✓ Bridges train-test distribution gap
  • ✓ Model practices error recovery
  • ✗ Too aggressive schedule → training instability
  • ✗ Argmax may still differ from sampling at inference
Scheduled sampling is like training wheels — you remove them gradually so the model learns to balance on its own before you let go completely.

Multi-Token Lookahead Training

Problem: autoregressive loss only optimizes "next token" — no long-range planning

Solution: Auxiliary heads predict downstream tokens from earlier hidden states

Lookahead pairs: (L1→L4), (L1→L6), (L3→L6), (L1→L7a)


Total Loss:

\[ \mathcal{L}_{\text{total}} = \mathcal{L}_{\text{main}} + \lambda \cdot \sum_{(\text{src}, \text{tgt})} \text{CE}\!\left(\text{head}_{\text{tgt}}(h_{\text{src}}),\; c_{\text{tgt}}\right) \]

where \(\lambda = 0.3\) (auxiliary loss weight)

Effect: Early tokens are forced to "plan ahead" for later levels.
The hidden state at L1 must encode enough information to predict L4, L6, etc.
Like a chess player thinking several moves ahead — the model at level 1 must already contain a "plan" for what tokens will come at levels 4, 6, 7. This reduces compounding errors.

Causal Pre-Training (GPT-style)

Phase 1: Pre-train

Objective: next-item-token prediction on ALL user histories (self-supervised)
  • Large item sequence corpus
  • Learns item co-occurrence patterns
  • Learns token transition probabilities
  • No labels needed

Phase 2: Fine-tune

Task-specific objective with purchase/conversion labels
  • Smaller supervised dataset
  • Alignment to business metrics
  • Fewer epochs needed (pre-trained representations)

Connection to GPT paradigm:

Internet text → task fine-tuning   ⟺   All sequences → recommendation fine-tuning

Pre-training on all interaction sequences is the "unsupervised internet" of recommendations — the model learns what items tend to follow what, before being told what's actually good.

RLHF / Reward Modeling for Recommendations

Standard RLHF Pipeline:

SFT → Reward Model → PPO

Mapping to GenRec:

SFT = pre-train with teacher forcing on interaction sequences
Reward = purchase / conversion / engagement signal
PPO = optimize generation policy toward high-reward items

Challenges:

  • Sparse reward: only know if user bought, not full preference ordering
  • Dense alternative: use CTR/CVR model as reward model
  • Reward hacking: model learns to generate only popular items
  • Instability: PPO training is notoriously unstable
RLHF says "generate item sequences that humans would actually buy." The challenge: purchase signal is binary and sparse, making reward modeling much harder than in text generation.

DPO — Direct Preference Optimization

Rafailov et al., 2023 — Eliminates the reward model entirely

DPO Loss:

\[ \mathcal{L}_{\text{DPO}} = -\log \sigma\!\left( \beta \left[ \log \frac{\pi_\theta(y_w|x)}{\pi_{\text{ref}}(y_w|x)} - \log \frac{\pi_\theta(y_l|x)}{\pi_{\text{ref}}(y_l|x)} \right] \right) \]

\(y_w\) = purchased item tokens (winner)   |   \(y_l\) = explored-but-not-purchased (loser)

\(\pi_{\text{ref}}\) = pre-trained model (frozen reference)   |   \(\beta\) = KL penalty strength

Two-Phase Training:

Phase 1: Pre-train with CE loss → πref
Phase 2: DPO fine-tune on preference pairs

Why DPO for GenRec:

  • No reward model instability
  • Natural preference pairs from logs
  • Shifts generation toward purchases
DPO: "Make the model more likely to generate items users bought, and less likely to generate items they saw but skipped." No reward model needed — just A/B comparison.

Contrastive Learning & InfoNCE

InfoNCE Loss (in-batch contrastive):

\[ \mathcal{L}_{\text{contrast}} = -\log \frac{\exp(h \cdot e_{\text{pos}} / \tau)}{\sum_{k=1}^{B} \exp(h \cdot e_k / \tau)} \]

\(h\) = decoder hidden state at end of generation

\(e_{\text{pos}}\) = correct item embedding   |   \(e_k\) = all items in batch (negatives)

\(\tau\) = temperature (low → hard negatives focus, high → smooth gradients)

Purpose:

Discriminative signal for collision resolution

Multiple items may share token prefixes — contrastive loss helps disambiguate

Complementary Roles:

Generation loss = recall (produce valid sequences)
Contrastive loss = precision (pick the RIGHT item)
Generation tells the model "produce valid token sequences." Contrastive learning says "among all valid sequences, make sure you're pointing at the specific item the user wants."

Knowledge Distillation

Transfer knowledge from a powerful teacher to an efficient student

Distillation Loss:

\[ \mathcal{L}_{\text{KD}} = \text{KL}(P_{\text{teacher}} \,\|\, P_{\text{student}}) = \sum_c P_T(c) \log \frac{P_T(c)}{P_S(c)} \]

Student learns to match the teacher's full distribution over tokens (soft targets)

Teacher Options:

  • Large pre-trained LLM
  • Ensemble of models
  • SASRec (discriminative) as teacher

Benefit:

Transfers knowledge from a model that "knows the right answer" to one that "generates the right answer"

Soft targets contain richer signal than hard labels (dark knowledge)

A discriminative model can tell you the probability of every item being correct. Distillation transfers that "ranking wisdom" into a generative model that must produce tokens one by one.

Curriculum Learning & Data Ordering

Idea: order training examples from easy → hard

Difficulty Dimensions:

EasyHard
Short historyLong history
Popular itemsTail items
Clear patternsAmbiguous intent
High purchase rateExploratory sessions

Schedule:

Epochs 1-N: train on easy sequences (high purchase rate, popular items)
Epochs N+1: gradually introduce hard examples (tail items, long sessions)

Connection to Graph Methods:

Same principle as negative sampling curriculum — start with easy negatives, add hard negatives later

You don't teach calculus on day one. Similarly, let the model master high-confidence patterns before challenging it with ambiguous tail-item sequences.

LoRA & Parameter-Efficient Fine-Tuning

Core idea: Freeze base weights \(W\), learn low-rank update:

\[ W' = W + \alpha \cdot A \cdot B \quad \text{where } A \in \mathbb{R}^{d \times r},\; B \in \mathbb{R}^{r \times d},\; r \ll d \]

Typical \(r = 4\text{-}64\) vs \(d = 768\text{-}4096\)

Parameters saved: from \(d^2\) → \(2 \cdot d \cdot r\) (e.g., 768² = 590K → 2×768×16 = 25K)

Application in GenRec:

Base model: pre-trained on full catalog
LoRA adapters: per-marketplace, per-category, per-campaign
  • One base model + small adapters for different contexts
  • Switch adapters at inference time (hot-swap)
  • Also applicable: adapter layers, prompt tuning, prefix tuning
LoRA says "the task-specific knowledge lives in a low-dimensional subspace." You don't need to retrain 100M parameters — just a 25K adapter captures the marketplace-specific behavior.

Label Smoothing & Token Dropout

Label Smoothing:

Soften targets from one-hot:

\[ y_{\text{smooth}} = (1-\varepsilon) \cdot \mathbf{1}_{\text{true}} + \frac{\varepsilon}{V-1} \cdot \mathbf{1}_{\text{others}} \]

Typical \(\varepsilon = 0.1\)

  • Prevents over-confidence on training tokens
  • Regularization that helps generalization
  • Especially helpful when token collisions exist

Token Dropout:

  • Input masking: randomly mask some history tokens during training
  • Forces robustness to incomplete history
  • Connection to BERT's MLM (but in decoder context)

Embedding Dropout:

  • Dropout on item embeddings during encoding
  • Prevents over-reliance on specific items
  • Typical rate: 0.1-0.3
Label smoothing says "don't be 100% sure about any single token — there might be valid alternatives." Token dropout says "don't rely on seeing every item in the history."

Speculative Decoding for Fast Inference

Leviathan et al., 2023 — Use a small draft model to accelerate a large target model

Algorithm:

1. Draft model (small, 2-layer transformer) proposes full sequence \([c_1, \ldots, c_K]\)

2. Target model verifies ALL tokens in a single parallel forward pass

3. Accept token \(c_t\) with probability:

\[ P_{\text{accept}}(c_t) = \min\!\left(1,\; \frac{P_{\text{target}}(c_t)}{P_{\text{draft}}(c_t)}\right) \]

If rejected → resample from adjusted distribution and discard remaining draft tokens

Why it works for GenRec:

  • Upper levels (L1-L4) are highly predictable
  • Draft model almost always correct there
  • Speedup concentrated in easy-to-predict levels

Key Properties:

Speedup: 2-4× for K=9 tokens
NO quality loss — exact same output distribution as target model
Speculative decoding is "optimistic parallelism" — guess the answer, then verify cheaply. For GenRec, upper tree levels are nearly deterministic, so the draft model's guesses are almost always accepted.

Diffusion-Based Distillation for Parallel Decoding

Goal: generate ALL tokens simultaneously via iterative denoising

Approach: Start from random/masked tokens → iteratively refine toward valid item tokens

Each denoising step refines ALL positions in parallel

After \(T\) steps (\(T \ll K\)): clean token sequence emerges

Distillation Pipeline:

1. Train autoregressive teacher (high quality, slow)
2. Train diffusion/masked student (parallel, fast)
3. Student matches teacher via KL divergence

Architecture Options:

  • Discrete diffusion (D3PM, Absorbing)
  • Continuous diffusion on embeddings → quantize
  • Masked parallel decoding (BERT-style)

Performance:

Speedup: 5-10× (T=3-5 steps vs K=9 sequential)
Quality: slight degradation (1-3% Recall drop)

Why feasible for GenRec:

Item tokens are SHORT sequences (K=4-9)

Parallel generation much more feasible than in long-form text (K=1000+)

Connection to MaskGIT / MDLM

Instead of generating tokens left-to-right, diffusion generates them all at once and refines iteratively. Short sequences (K=4-9) make this practical — you can't diffuse a novel, but you can diffuse a 9-token item code.

Alpha Sweep — Generation vs Discrimination Trade-off

Combined scoring at inference:

\[ \text{score} = \alpha \cdot P_{\text{generation}}(y|x) + (1-\alpha) \cdot \text{ID\_head\_score}(y|x) \]

Extreme Values:

αBehavior
1.0Pure generative (token probs only)
0.0Pure discriminative (ID head only)
0.3-0.5Sweet spot

Why the sweet spot works:

Generation provides diversity and novelty — explores the long tail
ID head provides precision — confirms the generated item is correct

Practical Sweep:

  • Evaluate Recall@K at α ∈ {0.0, 0.1, 0.2, ..., 1.0} on validation set
  • Typically: α=0.3-0.5 outperforms both endpoints by 5-15% relative
  • α can vary per domain (higher α for diverse categories, lower for head-heavy)
Generation and discrimination are complementary skills. The generation path says "these items are plausible completions," the ID head says "but THIS one is what the user actually wants." Mix them.

Quiz — Chapter 5

Q1: What is "exposure bias" in autoregressive generation, and why is it particularly dangerous for GenRec's hierarchical tokens?
A: Exposure bias is the mismatch between training (ground-truth prefix) and inference (model's own predictions as prefix). In GenRec, an error at level L1 propagates to ALL subsequent levels — the model has never practiced recovering from its own mistakes at upper hierarchy levels.
Q2: In DPO for GenRec, what constitutes a "winner" (y_w) and "loser" (y_l)? Why is DPO preferred over full RLHF?
A: Winner = item tokens the user purchased. Loser = item tokens the user explored but didn't purchase. DPO is preferred because it eliminates the reward model (no instability from RM training, no reward hacking), directly optimizing from naturally-occurring preference pairs in logs.
Q3: Multi-token lookahead uses pairs like (L1→L4). What does this mean concretely, and what λ value is used?
A: An auxiliary prediction head takes the hidden state at level 1 (hL1) and tries to predict the token at level 4 (cL4). This forces the L1 representation to "plan ahead." The auxiliary loss weight λ = 0.3 (30% of main loss magnitude).
Q4: In knowledge distillation for GenRec, why might a discriminative model (SASRec) make a good teacher for a generative student?
A: SASRec scores ALL items simultaneously and produces a probability distribution over the catalog. This "soft target" distribution is richer than hard labels — it tells the generative student not just WHICH item is correct but how much probability mass each alternative deserves (dark knowledge).

Quiz — Chapter 5 (continued)

Q5: LoRA learns ΔW = A·B with rank r. If d=768 and r=16, how many parameters does the adapter have vs full fine-tuning?
A: Full fine-tuning: d² = 768² = 589,824 parameters. LoRA adapter: 2×d×r = 2×768×16 = 24,576 parameters — a 24× reduction. This is per weight matrix (Q, K, V, etc.), so total savings are multiplicative across all layers.
Q6: Why does contrastive learning (InfoNCE) complement the generation loss? What does each address?
A: Generation loss optimizes recall — producing valid token sequences. Contrastive loss optimizes precision — disambiguating between items that share token prefixes (collisions). Together: the model generates valid sequences AND points to the correct specific item.
Q7: Speculative decoding gives 2-4× speedup with NO quality loss. What is the acceptance criterion for a draft token?
A: Accept draft token ct with probability min(1, Ptarget(ct) / Pdraft(ct)). If the target model assigns HIGHER probability than the draft model → always accept. If lower → accept proportionally. This guarantees the final distribution exactly matches the target model.
Q8: Diffusion-based distillation offers 5-10× speedup but with slight quality loss. Why is this approach particularly feasible for GenRec but not for general text generation?
A: GenRec item tokens are SHORT sequences (K=4-9 tokens). Parallel generation via diffusion/denoising is much more feasible for short sequences because: (1) fewer denoising steps needed, (2) inter-token dependencies are shorter-range, (3) 3-5 refinement iterations can cover K=9 positions. For text (K=1000+), diffusion quality degrades significantly.
