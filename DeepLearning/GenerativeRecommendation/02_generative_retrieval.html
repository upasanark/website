Chapter 2

The Generative Retrieval Paradigm

From scoring all candidates to generating items token-by-token — a new way to think about retrieval

What is Generative Retrieval?

Traditional (Discriminative) Retrieval

Score ALL \(N\) candidates → pick top-K
  • Items are atomic IDs
  • Model: \(f(user, item) \to score\)
  • Complexity: \(O(N)\) per query

Analogy: multiple-choice exam — pick from given options

Generative Retrieval

GENERATE the item token-by-token
  • Items are token sequences
  • Model: \(P(c_1, c_2, \ldots, c_K \mid context)\)
  • Complexity: \(O(K \cdot |V|)\) per item

Analogy: write-your-own-answer — construct the response

The key paradigm shift: items are no longer opaque IDs to be scored — they are TOKEN SEQUENCES to be generated. This changes everything about how retrieval systems are designed, trained, and scaled.

Why Generate Instead of Retrieve?

AdvantageExplanation
ScalabilityNo need to score all \(N\) items — just decode \(K\) tokens. Sublinear in catalog size.
CompositionalityTokens can recombine for novel items never seen in training.
InterpretabilityIntermediate tokens reveal model's reasoning path (e.g., category → brand → product).
Cold-startNew items get tokens from attributes — no need for interaction history.
Constrained generationEnforce business rules at decode time (availability, region, budget).
Trade-off: Harder optimization (exposure bias, token-level vs item-level loss mismatch), potentially lower raw accuracy on well-indexed catalogs, beam search artifacts.
Generative retrieval shines when the catalog is massive, dynamic, or needs constrained output. Discriminative methods still win when you need maximum precision on a small, static set.

The Encoder-Decoder Framework for GenRec

Encoder: Process user interaction history → context representation \(\mathbf{z}\)

\[ \mathbf{z} = \text{Encoder}(x_1, x_2, \ldots, x_T) \]

Decoder: Autoregressively generate item token sequence

\[ P(\text{item}) = P(c_1 \mid \mathbf{z}) \cdot P(c_2 \mid c_1, \mathbf{z}) \cdot \ldots \cdot P(c_K \mid c_1, \ldots, c_{K-1}, \mathbf{z}) \]

Architecture Variations

Encoder-Decoder
Separate modules for understanding history and generating output. (TIGER, DSI)
Decoder-Only
Single autoregressive model — history is part of the prompt. (GPT-style GenRec)
LLM-Based
Pretrained language models adapted for item generation. (P5, GPT4Rec)
Hybrid
Generate candidates → discriminative reranker. (Best of both worlds)
The math is just chain rule — decompose P(item) into a product of conditional token probabilities. The architecture choice determines how context is computed and how tokens are produced.

Autoregressive Generation — Step by Step

Given: user context \(\mathbf{z}\), vocabulary \(V\) of item tokens

Generate: item code \((c_1, c_2, \ldots, c_K)\)

Step 1: Predict first token — \(P(c_1 \mid \mathbf{z}) = \text{softmax}(W_1 \mathbf{z})\). Selects top-level category/cluster.
Step 2: Predict second token — \(P(c_2 \mid c_1, \mathbf{z})\). Refines within the chosen cluster.
Step 3: Continue until \(c_K\) — final token identifies the specific item.
Step 4: Beam search to keep top-B candidates at each step.

Example (4-level hierarchy):

\(c_1 = \text{"Electronics"}\) → \(c_2 = \text{"Audio"}\) → \(c_3 = \text{"Headphones"}\) → \(c_4 = \text{"Sony-WH1000XM5"}\)

Think of it like a librarian navigating a tree: first pick the section, then the shelf, then the book. Each token narrows the search space exponentially.

Key Design Dimensions

DimensionOptionsTrade-off
Tokenization Learned (VQ-VAE, RQ) vs Semantic (taxonomy, attributes) Accuracy vs Interpretability
Architecture Decoder-only vs Encoder-Decoder vs LLM-based Simplicity vs Expressiveness
Training Teacher forcing vs RL/DPO vs Contrastive Stability vs Optimality
Inference Greedy vs Beam search vs Constrained decoding Speed vs Quality
Evaluation uni100 (sample 100 negatives) vs Full-catalog ranking Comparable vs Realistic
These five dimensions define the design space of any generative recommendation system. Each choice interacts with the others — e.g., learned tokenization pairs best with teacher-forcing training, while semantic tokenization enables constrained decoding.
Every paper in this course makes specific choices along each dimension. Learning to identify these choices is key to understanding the field.

Generative vs Discriminative — When to Choose

Discriminative Wins When:

  • Full catalog is small (<100K items)
  • Latency is critical (<10ms)
  • Raw accuracy is paramount
  • Items are static (no cold-start)
  • ANN indexes are well-tuned

Generative Wins When:

  • Catalog is massive (>10M items)
  • Interpretability matters
  • Constrained generation needed
  • Cold-start is frequent
  • Novel item combinations possible

Hybrid Approach (best of both):

Stage 1: Generative model produces candidate set \(C\) (|C| ≈ 100–1000)

Stage 2: Discriminative reranker scores \(C\) for final top-K

Benefit: generative diversity + discriminative precision

In practice, most production systems will likely converge on hybrids — use generation for broad, interpretable retrieval and discrimination for precise final ranking. The question isn't "which paradigm" but "where in the funnel."

Quiz — Chapter 2

Q1: What is the fundamental representational shift in generative retrieval compared to traditional retrieval?
A: Items are represented as token sequences (not atomic IDs). This enables autoregressive generation — the model produces items token-by-token using the chain rule, rather than scoring a fixed set of candidates.
Q2: Write the factorized probability for generating a 4-token item code given user context \(\mathbf{z}\).
A: \(P(\text{item} \mid \mathbf{z}) = P(c_1 \mid \mathbf{z}) \cdot P(c_2 \mid c_1, \mathbf{z}) \cdot P(c_3 \mid c_1, c_2, \mathbf{z}) \cdot P(c_4 \mid c_1, c_2, c_3, \mathbf{z})\). Each token conditions on ALL previous tokens plus the user context.

Quiz — Chapter 2 (continued)

Q3: Name three advantages of generative retrieval over discriminative retrieval, and one disadvantage.
A: Advantages: (1) Scalability — no need to score all N items; (2) Cold-start handling — new items get tokens from attributes; (3) Constrained generation — business rules enforced at decode time. Disadvantage: Harder optimization — exposure bias, token-vs-item loss mismatch, and potentially lower raw accuracy on small static catalogs.
Q4: In a hybrid GenRec system, what role does the generative model play vs the discriminative model? Why combine them?
A: The generative model produces a diverse candidate set (broad retrieval from huge catalog, ~100–1000 items). The discriminative reranker scores these candidates for final top-K (precision). Combined: you get the scalability and compositionality of generation with the accuracy of point-wise scoring — generative diversity + discriminative precision.
