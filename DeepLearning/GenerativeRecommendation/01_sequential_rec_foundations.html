Chapter 1

Sequential Recommendation Foundations

From Markov chains to Transformers — how we model user behavior as sequences

The Sequential Recommendation Problem

Input

User's ordered interaction history:

\[ s = [i_1, i_2, \ldots, i_t] \]

Output

Predict next item \(i_{t+1}\)

Why Sequence Matters

  • Temporal patterns (weekday vs weekend)
  • Session context (browsing intent)
  • Intent evolution (explore → decide → buy)
"Order matters. A user who browsed headphones then a phone case has different intent than one who browsed phone case then headphones."

Evolution of Sequential Methods

"Each generation solved a limitation of the previous: fixed context → variable memory → parallel attention → full generation."

SASRec — Self-Attentive Sequential Rec

Kang & McAuley, 2018

Key Idea

Left-to-right causal self-attention over item ID embeddings

1. Embed each item: \(e_{i_k} \in \mathbb{R}^d\)
2. Stack \(L\) Transformer blocks with causal mask
3. Take final hidden state \(h_t\) as user representation
4. Score every item via dot product
\[ h_t = \text{Transformer}(e_{i_1}, \ldots, e_{i_t}) \] \[ \text{score}(i) = h_t \cdot e_i \]

Results (our data)

MetricValue
Recall@10.706
NDCG@100.838

BERT4Rec — Bidirectional Sequential Rec

Sun et al., 2019

Key Idea

Masked item prediction — like BERT's MLM but for items

1. Randomly mask items in the sequence
2. Predict masked items from both left and right context
3. Bidirectional attention — sees past AND future
\[ s' = [i_1, \texttt{[MASK]}, i_3, \ldots, i_t] \] \[ P(i_2 | s') = \text{softmax}(h_2 \cdot E^\top) \]

Cloze Task

At training: mask ~15% of items, predict them

At inference: append [MASK] at position \(t+1\), predict it

SASRec vs BERT4Rec

PropertySASRecBERT4Rec
AttentionCausal (L→R)Bidirectional

"BERT4Rec sees more context, but SASRec is naturally autoregressive — making it the backbone for generative approaches."

GRU4Rec — RNN-Based Sequential Rec

Hidasi et al., 2016

Architecture

GRU hidden state summarizes interaction history

\[ h_t = \text{GRU}(h_{t-1}, e_{i_t}) \] \[ \text{score}(i) = h_t \cdot e_i \]

Session-Based

Each user session treated as an independent sequence. Hidden state resets between sessions.

Results (our data)

MetricValue
Recall@10.628

Limitations

Vanishing gradients — struggles with long sequences
No parallelism — must process tokens sequentially
Fixed-size bottleneck — entire history compressed into one vector \(h_t\)
No direct attention to early items — information decays

The Prediction Problem — Softmax Over All Items

Standard Approach

Score every item in the catalog:

\[ P(i_{t+1} | s) = \text{softmax}(h_t \cdot E^\top) \]

where \(E \in \mathbb{R}^{|I| \times d}\) is the full item embedding matrix

Computational Cost

\(O(|I|)\) per prediction — score all items
\(|I| = 45\text{K}\) items → 45K dot products per user
At Amazon scale: \(|I| > 100\text{M}\) → intractable

What This Means

"Every sequential model — GRU4Rec, SASRec, BERT4Rec — faces this same bottleneck: scoring the entire catalog at inference time."

Limitations of ID-Based Methods

Vocabulary explosion: Each item needs a unique embedding vector — scales linearly with catalog size
Cold-start: New items have no learned embedding — impossible to recommend until retrained
No semantic meaning: Item ID 42 tells you nothing about the item itself
No compositionality: Can't combine "premium" + "headphones" + "Apple" to reason about unseen items
No generalization: Patterns learned for item A don't transfer to similar item B
"ID-based models memorize the catalog. They can't reason about items they haven't seen."

The Fundamental Issue

Item embeddings \(E \in \mathbb{R}^{|I| \times d}\)

  • \(|I| = 45\text{K}\) → 45K × 128 = 5.8M params just for items
  • \(|I| = 100\text{M}\) → 12.8B params — larger than the model!
  • New item → need to retrain or grow \(E\)
  • IDs are arbitrary: \(e_{42}\) and \(e_{43}\) share nothing

⚠️ This is the motivation for generative retrieval

From Discrimination to Generation

The paradigm shift — preview of Chapter 2

Discriminative (Classical)

Score all items, pick top-K

\[ \hat{i} = \arg\max_{i \in I} \; h_t \cdot e_i \]

Like a multiple-choice exam

  • ✓ Well-understood
  • ✓ High accuracy (SASRec R@1=0.706)
  • ✗ O(|I|) inference
  • ✗ Cold-start problem
  • ✗ Opaque item IDs

Generative (New Paradigm)

Generate the item token-by-token

\[ P(\text{item}) = \prod_{k=1}^{K} P(c_k | c_{Like writing your own answer

  • ✓ O(K·V) inference (K levels, V codebook)
  • ✓ Tokens carry semantic meaning
  • ✓ New items get tokens from attributes
  • ✓ Constrained generation is free
  • △ Harder optimization
"Instead of asking 'which of these 45K items?', we ask 'what kind of item?' and generate it piece by piece."

Quiz — Sequential Recommendation Foundations

Q1: In SASRec, why is the attention mask causal (left-to-right) rather than bidirectional?
A1: Because at inference time we only have past interactions — we can't peek at the future. Causal masking ensures the model trains in the same setting it will be evaluated: predicting the next item from history only. This also makes it naturally autoregressive, which is critical for generative extensions.
Q2: GRU4Rec achieves Recall@1 = 0.628 while SASRec achieves 0.706. What architectural difference explains this gap?
A2: SASRec uses self-attention, which allows direct connections between any two positions in the sequence (O(1) path length). GRU must propagate information step-by-step through hidden states, causing information from early items to decay. For sequences of length 20, GRU has path length 20 vs SASRec's path length 1.

Quiz — Sequential Recommendation Foundations (cont.)

Q3: The softmax prediction computes \(P(i_{t+1}|s) = \text{softmax}(h_t \cdot E^\top)\). If the catalog has 100M items and embeddings are 128-d, what's the computational cost per prediction and the memory for \(E\)?
A3: Compute: 100M dot products × 128 multiply-adds = 12.8 billion FLOPs per user prediction. Memory: \(100\text{M} \times 128 \times 4\text{ bytes} = 51.2\text{ GB}\) just for the item embedding matrix. This is why approximate methods (ANN, generative) are essential at scale.
Q4: A new product launches tomorrow. Under an ID-based model (SASRec) vs a generative model (GenRec), what happens?
A4: SASRec: The new item has no embedding in \(E\). It cannot be scored or recommended until the model is retrained — complete cold-start failure.
GenRec: The new item gets token assignments from its attributes (category, brand, price, intent). The model can immediately generate these tokens for relevant users. No retraining needed — the item is "born" with a valid representation.
