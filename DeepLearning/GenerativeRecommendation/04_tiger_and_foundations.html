TIGER & Foundational GenRec Models

From the first encoder-decoder generative recommender (NeurIPS 2023) through CoFiRec, LETTER, HiGR, Dual Identifiers, and GPR — the models that defined the field

TIGER — NeurIPS 2023

The First End-to-End Generative Retrieval for Recommendations

Encoder: SASRec-style self-attention over item embeddings from user history
Decoder: Autoregressive generation of RQ-VAE token sequences
Training: Teacher forcing on ground-truth token sequences
Inference: Beam search to generate top-K candidate items

Core factorization:

\[ P(\text{item} | \text{ctx}) = \prod_{k=1}^{K} P(c_k \mid c_{<k},\, \text{ctx}) \]

Loss (per level):

\[ \mathcal{L} = -\sum_{k=1}^{K} \log P(c_k^* \mid c_1^*,\ldots,c_{k-1}^*,\, h_{\text{enc}}) \]

where \(c_k^*\) = ground-truth token at level \(k\)

TIGER proved that generating item identifiers token-by-token can match or beat traditional retrieve-and-rank. The key insight: item tokens from RQ-VAE form a natural coarse-to-fine hierarchy perfect for autoregressive generation.

TIGER — Architecture Detail

History encoding: \(\; h_{\text{enc}} = \text{SelfAttn}(e_{i_1}, e_{i_2}, \ldots, e_{i_t})\)

Decoding step k: \(\; P(c_k) = \text{softmax}(W_k \cdot \text{CrossAttn}(c_{<k},\, h_{\text{enc}}))\)

The pipeline: each item gets a token representation (RQ-VAE), the user's history is encoded via attention, then the decoder generates the next item's tokens one level at a time — attending back to the full history at each step.

CoFiRec — CF Signal in Tokenization (Nov 2025)

Key Insight: Token Quality IS the Ceiling

Stage 1: Content-based clustering (RQ-VAE on item features)
Stage 2: Refine token assignments using collaborative filtering signal (co-interaction graph)
Result: Items co-purchased together get similar token prefixes

Two-stage objective:

\[ \mathcal{L}_{\text{CoFi}} = \underbrace{\mathcal{L}_{\text{RQ}}}_{\text{content}} + \lambda \underbrace{\mathcal{L}_{\text{CF}}}_{\text{collaborative}} \]

\(\mathcal{L}_{\text{CF}}\): encourage co-interacted items to share token prefixes

TokenizationRecall@10Δ vs Content-Only
RQ-VAE (content only)baseline—
CoFiRec (content + CF)+15–20%significant
"You can build the best decoder in the world, but if two different items share the same token sequence, the model physically cannot distinguish them. CoFiRec attacks the ceiling, not the architecture."

LETTER — LLM for Generative Retrieval (Li et al., 2024)

Bridge Between NLP and Recommendation

Core idea: Use a pre-trained LLM instead of training from scratch
Item representation: Text descriptions (title, attributes, reviews)
Generation: LLM generates textual item identifiers conditioned on user history
Advantage: Massive pre-trained knowledge about item semantics and language

Input format:

"User bought: [item₁ desc], [item₂ desc], ... → Recommend:"


Output: Generated item description

→ Resolved to catalog item via ANN search


AspectTIGERLETTER
BackboneTrained from scratchPre-trained LLM
Item IDsRQ-VAE codesText descriptions
Cold-startNeeds encoder passText is always available
LETTER asks: "Why learn item tokens from scratch when LLMs already understand language descriptions of products?" It trades token precision for the massive world knowledge embedded in pre-trained language models.

HiGR — Hierarchical Generative Retrieval (Tencent, Feb 2026)

Industrial-Scale Generative Retrieval

Challenge: Maintain retrieval quality at billion-item scale
Solution: Multi-level progressive refinement
Level 1: Coarse category generation (1000s of groups)
Level 2: Sub-category refinement within each group
Level 3: Fine-grained item selection

Hierarchical factorization:

\[ P(\text{item}) = \underbrace{P(\text{cluster})}_{\text{coarse}} \cdot \underbrace{P(\text{sub} | \text{cluster})}_{\text{mid}} \cdot \underbrace{P(\text{item} | \text{sub})}_{\text{fine}} \]

Each level has its own decoder head with progressively smaller candidate sets


ChallengeHiGR Solution
Codebook at 1B itemsHierarchical partitioning
Latency budgetProgressive pruning via beam
Token collisionsMore levels = fewer collisions
FreshnessAsync token re-assignment
HiGR proves generative retrieval can work at Tencent scale (billions of items). The trick: don't try to generate a single flat token sequence for billions of items — instead, progressively narrow the search space level by level.

Dual Identifiers (SIGIR-AP 2025)

Semantic + Collaborative IDs Together

The Problem:

  • Semantic tokens: capture content similarity but miss co-purchase signal
  • Collaborative tokens: capture behavioral patterns but lose interpretability
  • Neither alone is sufficient

The Solution:

Dual-head decoder: One head generates semantic tokens, another generates collaborative tokens
Fusion: Combined scoring from both heads at inference

Dual generation:

\[ P_{\text{sem}}(\text{item}) = \prod_k P(s_k | s_{<k}, \text{ctx}) \] \[ P_{\text{cf}}(\text{item}) = \prod_k P(t_k | t_{<k}, \text{ctx}) \]

Combined score:

\[ \text{score} = \alpha \cdot \log P_{\text{sem}} + (1-\alpha) \cdot \log P_{\text{cf}} \]
Why choose between "what the item IS" and "what the item is bought WITH"? Dual Identifiers generate both, then fuse. Semantic tokens handle cold-start; collaborative tokens handle behavioral nuance.

GPR — Generative Pretrained Recommender (Feb 2026)

GPT-Style Pre-training for Recommendation

Phase 1: Pre-train — GPT-style next-token prediction on massive interaction sequences
Phase 2: Fine-tune — Task-specific adaptation (next-item, campaign targeting, etc.)
Key idea: Learn universal item transition patterns from all users, then specialize

Pre-training objective:

\[ \mathcal{L}_{\text{GPR}} = -\sum_{t=1}^{T} \sum_{k=1}^{K} \log P(c_k^{(t)} \mid \text{seq}_{<t}) \]

Predict next item's tokens given all preceding item-token sequences


Fine-tuning:

+ task-specific loss (purchase prediction, engagement, etc.)

+ LoRA adapters for efficient multi-task

AspectTIGERGPR
Training paradigmEnd-to-end on target taskPre-train → fine-tune
Data requirementTask-specific onlyAll interaction logs (self-supervised)
TransferSingle taskMulti-task via adapters
Scale benefitModerateStrong (more data = better)
GPR applies the GPT playbook to recommendations: pre-train a foundation model on billions of interaction sequences, then fine-tune cheaply for any downstream task. The same insight that made GPT-3 powerful — scale pre-training data — works for item generation too.

Quiz — Chapter 4

Q1: TIGER factorizes item probability as \(\prod_k P(c_k | c_{<k}, \text{ctx})\). Why is this decomposition advantageous over predicting the full item ID in one shot?
A: (1) Reduces output vocabulary at each step from \(N\) items to \(V\) codebook entries (e.g., 256 vs 100K). (2) Enables beam search with progressive pruning — coarse predictions narrow the search space before fine-grained discrimination. (3) Hierarchical structure means early tokens can be constrained by business rules.
Q2: CoFiRec improves recall by 15–20% over content-only tokenization. What signal is it adding, and why can't the decoder learn this signal on its own?
A: CoFiRec adds collaborative filtering signal — co-purchase/co-interaction patterns — into the token assignment itself. The decoder can't compensate because if two frequently co-purchased items have different token prefixes (due to content-only clustering), the decoder has no path to associate them. Token assignment is a hard boundary the model cannot cross.

Quiz — Chapter 4 (continued)

Q3: Dual Identifiers uses both semantic and collaborative token heads. Why not simply make semantic tokens more fine-grained (more levels) to capture collaborative signal?
A: Semantic attributes (category, brand, price) are fundamentally content-based — they describe what an item IS, not what it's bought WITH. No amount of semantic granularity captures that "phone case buyers also buy screen protectors." Collaborative signal requires observing user behavior, which is orthogonal to item attributes. Dual heads let each modality optimize for its own signal without interference.
Q4: GPR pre-trains on all interaction sequences then fine-tunes. What's the risk of this approach compared to TIGER's end-to-end training?
A: (1) Distribution mismatch — pre-training sees ALL interactions (browsing, clicks, purchases) but the fine-tuning task may only care about purchases. The model may have learned token transitions that optimize for engagement, not conversion. (2) Catastrophic forgetting — aggressive fine-tuning may overwrite useful pre-trained knowledge. (3) Compute cost — pre-training on billions of sequences is expensive upfront, only justified if multiple downstream tasks amortize the cost.
