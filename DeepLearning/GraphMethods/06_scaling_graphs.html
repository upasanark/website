MapReduce for Graph Propagation

Core insight: Message passing on graphs = Map + Reduce

Map phase — for each edge \((u, i)\), emit:

\[ \text{emit}\bigl(i,\; w_{ui} \cdot \mathbf{e}_u\bigr) \]

Reduce phase — for each node \(i\), aggregate incoming messages:

\[ \mathbf{e}_i^{\text{new}} = \text{normalize}\!\left(\sum_{\text{messages to } i} m_j\right) \]

This exactly implements:

\[ \mathbf{e}_i^{(l+1)} = \sum_{j \in \mathcal{N}(i)} \frac{w_{ji}}{d_i} \cdot \mathbf{e}_j^{(l)} \]

Multi-hop: chain K MapReduce jobs = K propagation layers

One MapReduce job = one propagation layer. The shuffle phase groups all messages by destination node — exactly what aggregation needs.

Edge-Wise Partitioning for Scalable Propagation

Strategy: partition edges (not nodes) across workers

Partition edges into P sets: \(\; E = E_1 \cup E_2 \cup \ldots \cup E_P\)


Worker \(p\) computes partial aggregation:

\[ \text{partial}_i^{(p)} = \sum_{\substack{j:\; (j,i) \in E_p}} w_{ji} \cdot \mathbf{e}_j \]

Final merge:

\[ \mathbf{e}_i^{\text{new}} = \text{normalize}\!\left(\sum_{p=1}^{P} \text{partial}_i^{(p)}\right) \]

Why this works:

  • Edges are independent during message phase
  • Addition is associative → partial sums are valid
  • Linear scaling with number of edges

Production usage:

  • 423M edges ÷ 30 executors
  • ≈ 14M edges per worker
  • Partial sums merge at reduce step
Since summation is associative, we can split the sum across workers and combine later. Each worker handles its edge slice independently — embarrassingly parallel during the message phase.

Spark/PySpark Implementation Pattern

# One propagation layer in PySpark
edges_df = spark.read.parquet("edges/")       # (src, dst, weight)
embeddings_df = spark.read.parquet("embeddings/")  # (node_id, embedding)

# MAP: join edges with source embeddings, compute messages
messages = (edges_df
    .join(embeddings_df, edges_df.src == embeddings_df.node_id)
    .select("dst", (col("weight") * col("embedding")).alias("message"))
)

# REDUCE: aggregate messages at each destination
new_embeddings = (messages
    .groupBy("dst")
    .agg(array_sum("message").alias("aggregated"))
    .withColumn("embedding", normalize_udf("aggregated"))
)

Key Optimizations:

Broadcast small tables (item embeddings if |I| < 10M rows)
Repartition edges by destination for efficient reduce
Cache intermediate DataFrames across propagation layers
Coalesce output to avoid small-file problem
The Spark DAG mirrors the mathematical formulation exactly: join = attach source embeddings, groupBy = aggregate by destination. The optimizer handles the parallelism.

Distributed FAISS for Similarity Search

The Problem:

150M users × 256 dims × 4 bytes

= 153 GB of user embeddings


Need: top-K items for each user

\[ \text{score}(u, i) = \frac{\mathbf{e}_u \cdot \mathbf{e}_i}{\|\mathbf{e}_u\| \cdot \|\mathbf{e}_i\|} \]

FAISS Index Types:

IndexSpeedExactness
IndexFlatIPSlowExact
IndexIVFFastApproximate

Distribution Strategy:

Performance: ~2,500–3,000 users/sec → ~60 min for 150M users on 30 executors
Broadcast the small item index to every worker, partition users across workers. Each worker queries its user partition against the local copy of the index — zero network traffic during search.

Memory Management & Broadcast Variables

What to Broadcast:

DataSize
FAISS index (4K items × 256d)~50 MB
Item metadata~100 MB
Cluster assignments~10 MB

Partition-Level Processing:

Rebuild index once per partition, process all rows in that partition
Anti-pattern: shipping large data per-row → serialization overhead kills performance

Memory Budget per Executor (6–8 GB):

Broadcast once, use everywhere. The key insight: send the small table (items) to all workers rather than shuffling the large table (users) to one place.

Practical Challenges — Data Skew & Shuffle

The Skew Problem:

Popular items have millions of edges (avg: 100). One reducer gets 10,000× more work.

Solution: Salting — two-stage aggregation


Stage 1: partial_agg by \((\text{item\_id} \,\|\, \text{salt})\) where salt ∈ {0..S-1}

→ Distributes the hot key across S reducers


Stage 2: final_agg by \(\text{item\_id}\) alone

→ Merges S partial results (now balanced)

Other Strategies:

Broadcast join: when one side < 10 GB, avoid shuffle entirely
Checkpointing: save every layer — if layer 3 fails, restart from layer 2 (not scratch)
# Salting example
salted = edges.withColumn("salt", (rand() * NUM_SALTS).cast("int"))
stage1 = salted.groupBy("dst", "salt").agg(sum_vectors("message"))
stage2 = stage1.groupBy("dst").agg(sum_vectors("partial_sum"))
Data skew is the #1 production killer for graph workloads. Salting turns one overloaded reducer into S balanced reducers at the cost of one extra shuffle — almost always worth it.

Quiz — Chapter 6

Q1: For a graph with 100M edges and 30 executors using edge-wise partitioning, how many edges does each executor process?
A: 100M ÷ 30 ≈ 3.33M edges per executor. Each worker computes partial aggregations for its edge subset, then partial results are merged per destination node.
Q2: Why broadcast the FAISS index instead of the user embeddings?
A: The item index is small (~4K items × 256 dims = 50 MB) while user embeddings are huge (150M × 256 = 153 GB). Broadcasting the small side avoids shuffling the large side. Each worker queries its local users against the broadcast index with zero network traffic.
Q3: In MapReduce graph propagation, the Map phase emits (destination, message). What does the Reduce phase compute?
A: The Reduce phase aggregates all incoming messages for each destination node: \(\mathbf{e}_i^{\text{new}} = \text{normalize}(\sum_j m_j)\). This implements the neighborhood aggregation step of propagation.

Quiz — Chapter 6 (continued)

Q4: A popular item has 5M incident edges while average items have 100. How does salting help?
A: Without salting, one reducer handles all 5M edges for the popular item (50,000× slower than average). Salting with S=100 splits this into 100 partial aggregations of 50K edges each (balanced). A second stage merges the 100 partial results — now the max reducer load is only 500× not 50,000×.
Q5: You have 150M user embeddings × 256 dimensions in float32. How much storage is that?
A: \(150\text{M} \times 256 \times 4\text{ bytes} = 150 \times 10^6 \times 1024\text{ bytes} \approx\) 153 GB. This is why user embeddings must be partitioned across workers rather than fit in a single machine's memory.
