Section A: Fundamentals (Q1–Q2)

Q1: Explain message passing in GNNs in one sentence. Why is it the unifying framework?
Each node iteratively collects and aggregates information from its neighbors, then updates its representation. It unifies GCN, GraphSAGE, GAT — they differ only in how they aggregate (mean vs attention vs sampled).
Q2: What is the Graph Laplacian and why does it matter for GNNs?
\(L = D - A\). It measures local smoothness — its eigenvectors define the "frequency" basis for graphs (like Fourier for signals). Key property: \(x^T L x = \sum_{(i,j)\in E}(x_i - x_j)^2\) measures total signal variation. GCN's spectral derivation starts here.

Section A: Fundamentals (Q3–Q4)

Q3: What is over-smoothing? Give the mathematical intuition.
After K layers of mean aggregation, all embeddings converge to the same vector. Repeated multiplication by \(\hat{A}\) = low-pass filter killing discriminative components. After \(K\to\infty\): \(h_i^{(K)} \to d_i/2m\). Practical: GNNs rarely benefit from >3 layers.
Q4: Why does GCN use \(\tilde{D}^{-1/2}\tilde{A}\tilde{D}^{-1/2}\)? What goes wrong with just \(A\)?
Without normalization, high-degree nodes dominate (features summed more). Symmetric normalization scales by \(1/\sqrt{d_i \cdot d_j}\) — equal "voting power" regardless of degree. Self-loop (\(\tilde{A}=A+I\)) ensures a node retains its own features.

Section A: Recommendations (Q5–Q6)

Q5: Why does LightGCN remove nonlinearities and transformation matrices?
For CF, nodes have no meaningful features (just IDs). Transform \(W\) on random embeddings = different random init. Nonlinearity adds noise since collaborative signal IS linear structure flow. Empirically: LightGCN > NGCF > GCN on all CF benchmarks.
Q6: How does XGCN differ from end-to-end GNN training? What's the trade-off?
XGCN freezes propagation (forward only, no gradient through graph) and trains an MLP on propagated embeddings. Much cheaper (no sparse backprop), but can't jointly optimize. Works well because graph structure is a good prior. Fails when graph is noisy or task needs learned edge importance.

Section A: Recommendations (Q7–Q8)

Q7: Design a GNN for cold-start recommendations. Walk through your approach.
(1) Hybrid graph: interactions + KG attribute edges. (2) New items connect via attributes from day one. (3) GraphSAGE for inductive embedding. (4) Category-level fallback. (5) λ blending: sigmoid(interactions/scale − shift). (6) Hard negatives from same category.
Q8: What's the relationship between MF and GNNs?
MF = 1-layer LightGCN. \(u = \sum_{i\in N(u)} \frac{1}{\sqrt{d_u}\sqrt{d_i}} e_i\) = one normalized aggregation pass on bipartite graph. LightGCN extends to K layers (K-hop collaborative signal). All GNNs reduce to MF when L=1, no transforms.

Section A: Advanced (Q9–Q10)

Q9: Over-squashing vs over-smoothing — what's the difference?
Over-smoothing = representations become indistinguishable (all converge). Over-squashing = distant information compressed through bottleneck nodes (\(\partial h_v^{(K)}/\partial x_u\) decays exponentially). Both limit depth but for different structural reasons.
Q10: How does Graphormer encode graph structure without message passing?
(1) Centrality encoding: degree as input bias. (2) Spatial encoding: shortest-path distance as attention bias. (3) Edge encoding: edge features along shortest path averaged into attention. The Transformer "knows" topology via these biases — no explicit propagation.

Section A: Advanced (Q11–Q12)

Q11: Design a GNN for a heterogeneous graph (multiple node/edge types).
R-GCN: \(h_i' = \sigma(\sum_r \sum_{j\in N_r(i)} \frac{1}{|N_r|} W_r h_j)\). Per-relation weight \(W_r\). Problem: \(|R| \times d^2\) parameters. Solutions: basis decomposition \(W_r = \sum_b a_{rb} B_b\), or block-diagonal \(W_r\).
Q12: Complexity comparison — GCN vs GAT vs Graph Transformer.
GCN: \(O(|E|\cdot d + |V|\cdot d^2)\) — sparse. GAT: \(O(|E|\cdot d)\) — still sparse. Graph Transformer: \(O(|V|^2\cdot d)\) — dense attention. For 28M nodes: GCN/GAT feasible, Transformer impossible without subgraph sampling.

Section A: System Design (Q13–Q14)

Q13: Design a real-time graph-based recommendation system.
Offline (daily/weekly): full GNN training + embedding computation on distributed cluster. Online (real-time): user embedding lookup → FAISS top-K (< 10ms) → lightweight reranking. Near-real-time: incremental embedding updates for active users.
Q14: How would you A/B test a GNN recommender against a CF baseline?
Randomize users into treatment/control. Primary metrics: CTR, conversion, revenue. Secondary: diversity, coverage. Guardrails: satisfaction, returns. Power analysis for sample size. Minimum 4 weeks. Segment analysis: cold-start users vs heavy users.

Section A: System Design (Q15–Q16)

Q15: Great offline metrics but poor A/B results — diagnose.
(1) Distribution mismatch: easy offline negatives. (2) Temporal leakage in train/test. (3) Popularity bias: recommends items user finds anyway. (4) Latency → fallback served. (5) Low diversity: similar items bore after first click. (6) Feedback loop favoring baseline.
Q16: Serve personalized recs for 150M users — infrastructure.
Embedding KV store (Redis/DynamoDB). Item index on FAISS serving instances. Flow: fetch embedding (1ms) → ANN query (5ms) → rerank (3ms) = ~10ms total. Weekly full recompute, daily incremental. Shard users by hash, replicate item index. Fallback: category popular items.

Section B: Method Selection Decision Tree

Need collaborative signal?
├── No  → Content-based or sequential model
└── Yes → GNN-based
    │
    ├── Scale > 10M nodes?
    │   ├── Yes → XGCN (frozen propagation) or LightGCN (mini-batch)
    │   └── No  → LightGCN / GAT (full-batch feasible)
    │
    ├── Multi-interest needed?
    │   ├── Yes → Clustered propagation (explicit) or DGCF (latent)
    │   └── No  → Single embedding per user
    │
    ├── Cold-start items?
    │   ├── Yes → Hybrid: KG attribute edges + interaction graph
    │   └── No  → Pure interaction graph sufficient
    │
    └── Real-time updates?
        ├── Yes → Incremental embedding updates + streaming edges
        └── No  → Daily/weekly batch recompute
Start with simplest method that satisfies your constraints. Complexity should be justified by measurable metric gains.

Section C: Common Interview Patterns

Pattern 1: "Design a recommendation system for X"
Framework: Data → Graph Construction → Method Selection → Training → Serving → Evaluation
Always mention: scale, cold-start, latency, evaluation metrics. Graph angle: nodes? edges? weights?
Pattern 2: "How would you improve metric X?"
Framework: Current approach → Diagnosis → Hypothesis → Solution → Measurement
Graph solutions: add edge types, more layers, multi-interest, temporal features
Pattern 3: "Explain concept X at multiple levels"
Level 1 (intuition): one-sentence analogy. Level 2 (technical): math + architecture. Level 3 (practical): trade-offs, failure modes.
Pattern 4: "Debug this system"
Framework: Symptoms → Hypotheses → Diagnostics → Root cause → Fix → Validation
Common GNN issues: over-smoothing, embedding collapse, data leakage, popularity bias

Section C: Deep Dive Follow-ups

Why GMM over K-means?
Soft assignments allow items to contribute to multiple clusters with varying probability. An electronics item can be 60% "tech" and 40% "gifts" — hard assignment loses this nuance.
Why normalize per layer?
Prevents dominant clusters (with more items/edges) from absorbing smaller niche interests during propagation. Without it, the "popular" cluster overwhelms minority interests by layer 3.
Why re-anchor embeddings?
Pre-trained embeddings encode item semantics. Propagation should ADD collaborative signal, not REPLACE semantic signal. Re-anchoring = residual connection: \(e_{final} = \alpha \cdot e_{propagated} + (1-\alpha) \cdot e_{pretrained}\).
What would you do differently?
(1) Learned edge weights (MLP on edge features) instead of fixed α. (2) Attention-based cluster assignment instead of GMM. (3) Online A/B testing earlier in development. (4) Temporal edge decay for freshness.
