Graph Neural Networks & Graph Methods

What does a GNN capture that a feedforward network on tabular features cannot? Use the wind farm wake example. ▶
Tabular models treat each sample as i.i.d. — each turbine is an independent row. A GNN encodes relational structure: upstream turbines create wake effects on downstream ones, and multi-hop propagation captures cascading chains (A wakes B wakes C). Edge features can carry direction, distance, and wake intensity. The graph also adapts dynamically with wind direction — something hard to express in a fixed feature table.
Why does LightGCN remove both W (weight matrix) and σ (nonlinearity) from the standard GCN layer for collaborative filtering? ▶
In CF, user/item nodes start with random ID embeddings — no meaningful input features. Applying W to a random vector just produces a different random vector. Nonlinearity between layers adds complexity with no benefit when the signal is purely structural. The key insight: for CF, the graph IS the signal. The only useful operation is neighborhood averaging. LightGCN pushes all learning into the embeddings themselves, optimized so that when transformed by the normalized adjacency matrix, the result satisfies the CF objective. Parameters scale O(n × d) regardless of graph density.
What is over-smoothing in GNNs? Why are most GNNs limited to 2–3 layers? ▶
After K layers of mean aggregation, all node embeddings converge toward the global mean. Mathematically, repeated averaging is a low-pass filter that kills discriminative signal. After 2–3 layers, nodes become indistinguishable. Mitigations: Residual connections (h' = h + GNN(h)), Jumping Knowledge (combine all layer outputs via concat/max/attention), DropEdge (randomly remove edges during training to slow diffusion).
For a graph with 28M user nodes, full self-attention is O(n²). Name two strategies to make graph transformers tractable. ▶
1. Sparse attention: Only attend to graph neighbors + sampled distant nodes. Trade-off: loses some long-range signal but keeps O(n·k) complexity.

2. Cluster-based attention: Cluster nodes, attend within clusters + across cluster representatives. Trade-off: introduces quantization error at cluster boundaries, but enables hierarchical global context.

Other options: linear attention (Performer — approximate softmax with random features → O(n)), subgraph sampling (ClusterGCN style).
In GAT vs GCN — what's the fundamental difference in how neighbors are weighted, and when does GAT help? ▶
GCN uses fixed weights from graph structure (degree normalization: 1/√(dᵢ·dⱼ)). All neighbors contribute equally (modulo degree). GAT learns attention coefficients: αᵢⱼ = softmax(LeakyReLU(aᵀ[Whᵢ ∥ Whⱼ])). GAT helps when not all neighbors are equally relevant — e.g., in a user-item graph where some interactions are noise (accidental clicks) and others are strong signal (purchases). Multi-head attention lets the model attend to different aspects simultaneously.
How would you handle the multi-interest problem in graph-based recommendations? Why does a single user embedding fail? ▶
A user who buys electronics + yoga gear + cookbooks gets a single embedding that's close to none of those clusters — the "average user" fallacy. Solutions: DGCF (disentangled channels, but latent and expensive), PIMIRec (prototype clustering, but no collaborative signal), or explicit clustering — GMM cluster item embeddings, tag edges with cluster identity, propagate per-cluster to get K interest-specific embeddings per user. Score candidate items against all user prototypes and take max. The explicit approach is interpretable, compatible with pre-trained embeddings, and scales well.
In your wind farm work, would the turbine graph be directed or undirected, homogeneous or heterogeneous? Why? ▶
Directed — wake effects are asymmetric: an upwind turbine impacts a downwind turbine but not vice versa. Homogeneous — all nodes are the same entity type (turbines) with the same feature space. The edge direction (and existence) depends on current wind direction, so you'd either rebuild the adjacency per condition, use edge features encoding directional wind alignment, or use attention (GAT-style) to let the model learn which neighbors matter given current conditions.

Generative Recommendation & Sequential Models

SASRec uses causal attention, BERT4Rec uses bidirectional. Why can't you directly deploy BERT4Rec for next-item prediction? ▶
BERT4Rec trains with Cloze task — masking items in the middle with context on both sides. At inference, you append [MASK] at position t+1, but now there's nothing to the right — train-test distribution mismatch. The model never practiced predicting from one-sided context. SASRec's causal masking means position t never sees future positions during training, so next-item prediction at test time exactly matches the training task. No distribution shift.
What are the three fundamental limitations of ID-based sequential recommenders (SASRec, BERT4Rec)? ▶
1. Cold start: New items have no learned embedding — zero signal until interactions accumulate.

2. No semantics: ID 42 tells you nothing about the item — all relationships must be learned from co-occurrence data alone.

3. Vocabulary scale: Softmax over millions of items is computationally expensive and each item needs a separate full-dimensional embedding vector. No parameter sharing between similar items.
How does TIGER's semantic tokenization solve cold-start, and what's the key assumption that makes it work? ▶
TIGER assigns each item a token sequence [t₁, t₂, t₃] derived from item features and collaborative info via RQ-VAE. Cold-start items get tokens based on their features alone — no interactions needed.

Key assumption: Semantically similar items share token prefixes. If two running shoes are similar, they might be [12, 45, 78] and [12, 45, 93] — same prefix, different final token. This is like a zip code: "943xx" = Palo Alto area. The RQ-VAE is trained to ensure nearby items in embedding space map to nearby paths in the codebook tree. If this property breaks, autoregressive generation can't generalize.
In beam search for generative retrieval, what's the failure mode of greedy decoding (beam=1) and why is it worse for recommendations than for text? ▶
Greedy picks the highest-probability t₁, then best t₂ given that t₁. But the globally best item might have a mediocre first token. A single wrong greedy choice at t₁ locks you out of the entire correct subtree.

Why worse for recommendations than text: In text, many continuations are acceptable — greedy is "good enough." In recommendation, token sequences must exactly match valid items in the catalog. [12, 45, 78] is a specific product; [12, 45, 79] is completely different. There's no "close enough." Also, recommendations need a ranked top-K list, not just one answer — beam search naturally produces K candidates with scores.
What is RQ-VAE and why is it preferred over a flat VQ-VAE for item tokenization? ▶
VQ-VAE maps each item to a single code from one codebook of size K — only K distinct items can be represented, or you need a massive codebook.

RQ-VAE (Residual Quantization) uses a sequence of codebooks. The first codebook quantizes the embedding, then the residual (error) is quantized by the second codebook, and so on. With L codebooks of size K, you can represent K^L items. This gives a natural hierarchy: early tokens capture coarse semantics, later tokens refine — exactly what autoregressive generation needs. It's also more parameter-efficient than a single codebook of size K^L.
How would you evaluate a generative retrieval system vs a traditional two-tower retrieval model? ▶
Metrics: Recall@K, NDCG@K, Hit Rate — same as traditional retrieval.

Key differences in evaluation:
- Generative models may produce invalid token sequences (sequences that don't map to any real item) — measure "valid generation rate"
- Latency comparison: two-tower uses ANN search (fast), generative uses beam search (potentially slower with large beams)
- Cold-start split: test on items with zero training interactions — generative should excel here
- Diversity: does beam search produce diverse results or mode-collapse to similar items sharing prefixes?

General ML & Applied Science

BPR loss vs InfoNCE loss for learning embeddings — what's the difference and when do you prefer each? ▶
BPR (Bayesian Personalized Ranking): Pairwise — L = -Σ ln σ(ŷ_ui - ŷ_uj). Requires one positive, one sampled negative per update. Simple, well-understood gradients.

InfoNCE: Listwise — L = -log(exp(sim(u,i)/τ) / Σₖ exp(sim(u,k)/τ)). Uses all negatives in the batch denominator. More negative signal per step, but sensitive to temperature τ.

Prefer BPR when compute budget is tight or negatives are carefully curated. Prefer InfoNCE when you can use large batch sizes (in-batch negatives are free) and want stronger contrastive signal. InfoNCE generally works better with more negatives, BPR is more stable with fewer.
You built surrogate models for wind turbine load prediction. How do you decide between a Gaussian process and a neural network as the surrogate? ▶
Gaussian Process: Gives calibrated uncertainty estimates out of the box. Great for small data (< 10K points), sequential optimization (Bayesian optimization), and when you need to know how confident the prediction is. Scales O(n³) naively.

Neural Network: Scales to large datasets, handles high-dimensional inputs and complex nonlinearities better. But uncertainty requires extra work (ensembles, MC dropout).

In the wind farm context: GPs were ideal for wake modeling (relatively low-dimensional input: wind speed, direction, yaw angles; moderate data sizes per farm configuration; uncertainty needed for robust optimization). NNs were better for load prediction across full design spaces with many parameters and large DOE datasets.
What's the difference between hard negatives and random negatives in contrastive learning, and what's the risk of each? ▶
Random negatives: Sample uniformly from all items. Simple but often "too easy" — the model learns nothing from items the user clearly wouldn't interact with (trivially distinguishable).

Hard negatives: Items close in embedding space but not interacted with. Provide strong gradient signal but risk false negatives — the user might actually like the item but just hasn't seen it yet. Training on false negatives teaches the model wrong boundaries.

Sweet spot: Popularity-biased sampling (popular items user chose NOT to interact with) or semi-hard negatives (close but not closest). In-batch negatives offer a natural difficulty distribution without extra sampling cost.
How does your real-time anomaly detection on turbine sensor signals differ from anomaly detection in a recommendation system? ▶
Turbine signals: Time-series, high-frequency (kHz), physics-governed. Anomalies are deviations from physical models — sensor faults, mechanical degradation. Ground truth exists (component failures are recorded). Methods: signal processing (FFT, wavelets), reconstruction error (autoencoders), statistical process control.

Recommendation systems: Anomalies are behavioral — bot traffic, click fraud, sudden popularity spikes. Data is sparse, event-driven (not continuous), and ground truth is rare. Methods: session pattern analysis, deviation from user's own history, graph-based anomalies (sudden degree changes).

Key shared principle: Both define "normal" via a model and flag deviations. The difference is the signal structure (continuous physics vs discrete interactions) and the evaluation (precision/recall on labeled faults vs business-metric impact).
You've worked across wind energy, healthcare (ventilators), and recommendation systems. How does "deploying ML in production" differ across these domains? ▶
Wind energy: Models run on edge/SCADA systems with limited compute. Latency requirement is seconds (control loop). Safety-critical — a bad prediction can damage equipment worth millions. Extensive simulation testing before field deployment. Regulatory compliance.

Healthcare (ventilators): Most stringent — FDA/regulatory approval required. Patient safety is paramount. Models must be interpretable to clinicians. Real-time (milliseconds for breath detection). Extensive clinical validation with diverse patient populations.

Recommendations: Cloud-deployed, abundant compute. Latency target ~100ms (user-facing). Not safety-critical but economically important. A/B testing is easy — deploy to 1% of traffic. Can iterate fast. Model retraining is frequent (daily/weekly) as user behavior shifts.

Common thread: All require monitoring for drift, but the consequences of failure and the iteration speed differ by orders of magnitude.
Explain the bias-variance trade-off in the context of choosing model complexity for a surrogate model. ▶
High bias (too simple): A linear surrogate for turbine load can't capture nonlinear aerodynamics — systematically underpredicts at extreme conditions. Low training error variance but consistent error.

High variance (too complex): A very deep network fits training DOE points perfectly but oscillates wildly between them — bad extrapolation in unexplored design regions.

In surrogate modeling specifically: The design space is often sparsely sampled (expensive simulations). High-variance models give false confidence — the surrogate says "this design is optimal" but it's just overfitting noise in the DOE. GPs with appropriate kernels offer a natural regularization (kernel smoothness) and uncertainty that grows in data-sparse regions — naturally communicating where the bias-variance trade-off is uncertain.
What is the difference between offline evaluation and online A/B testing for recommendation systems? When can offline metrics be misleading? ▶
Offline: Hold out recent interactions, predict them from history. Metrics: Recall@K, NDCG@K, Hit Rate. Fast, cheap, reproducible.

Online A/B: Deploy to real users, measure business metrics (CTR, revenue, engagement, retention). Ground truth from actual behavior.

Offline can be misleading because:
- Popularity bias: Models that recommend popular items score well offline (more positive labels) but don't help users discover new items
- Missing data: Items not shown can't be interacted with — absence of interaction ≠ negative preference
- Position bias: Users click higher-ranked items more regardless of relevance
- Feedback loops: The model that generated the data influences what interactions exist

Rule of thumb: offline for rapid iteration and filtering bad models; online for final decisions.
